Telling hashes apart by sight alone

A hash is just a string — nothing in it says "I am MD5." identify reasons from three observable signals: prefix, length, and charset. Here is every format it recognizes, and exactly what separates them.

01 The one decision that splits everything

Before counting anything, ask a single question: does the string label itself? That fork decides which signal you read next.

starts with $ or {

Structured / self-labeling

Uses Modular Crypt Format — $id$params$salt$digest. The $id$ names the scheme outright.

→ read the prefix · high confidence
all hex 0-9 a-f

Bare digest

No markings at all — just raw hex. The only thing to go on is how many characters there are.

→ count the length · sometimes ambiguous

02 Bare digests — classified by length

Every hex digit encodes 4 bits, so hex length = output bits ÷ 4. SHA-256 → 256/4 = 64 chars. That ratio is the whole logic. Where a length is shared, identify returns ranked guesses rather than a false certainty.

The unbreakable tie: MD5 and NTLM are both 32 hex chars with no prefix — byte-for-byte indistinguishable. Only context decides: a web-app DB leak leans MD5; a Windows secretsdump leans NTLM. That's why a 32-hex string returns both, ranked.

04 The distinction that actually matters

You asked how SHA-512 differs from sha512crypt. They share a number but live in opposite worlds — and this fast-vs-slow split is the real security story behind the whole table.

● fast digests — wrong tool for passwords

Built for speed

General-purpose checksums. A GPU computes billions per second — which is exactly why they're a disaster for storing passwords, and fine for file integrity.

  • MD5, SHA-1, SHA-256, SHA-512
  • NTLM (raw MD4, unsalted)
  • no salt, no cost knob, no defense against brute force

● password hashes — built to resist

Slow on purpose

Designed to be expensive: salted, with a tunable cost (or memory-hardness) that blunts the GPU's advantage. This is what a password column should hold.

  • bcrypt — tunable cost factor
  • sha512crypt / sha256crypt — thousands of rounds + salt
  • argon2 / yescrypt — memory-hard, modern best practice
So: SHA-512 is a raw 512-bit digest — fast, no salt, 128 hex chars, no prefix. sha512crypt merely uses SHA-512 inside a deliberately slow, salted, thousands-of-rounds construction — and labels itself $6$. Same building block, opposite purpose.
